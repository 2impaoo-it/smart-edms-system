Đồ án J2EE: 
1. Tên Đề tài & Mô tả Tổng quan
Tên đề tài: Xây dựng Hệ thống Quản lý Tài liệu và Quy trình Ký số Nội bộ (Smart E-DMS)

Mô tả: Đây là một giải pháp phần mềm dành cho doanh nghiệp nhằm loại bỏ văn bản giấy tờ truyền thống. Hệ thống cho phép số hóa việc lưu trữ, quản lý phiên bản tài liệu và đặc biệt là thực hiện quy trình "Trình ký online". Thay vì phải in hợp đồng ra giấy và cầm đi xin chữ ký từng sếp, nhân viên chỉ cần upload lên hệ thống, các cấp lãnh đạo sẽ ký duyệt bằng Chữ ký số bảo mật ngay trên trình duyệt.

Mục tiêu cốt lõi:

Lưu trữ tập trung: Không còn cảnh file nằm rải rác ở máy cá nhân.

Bảo mật: Đảm bảo tài liệu không bị sửa đổi sau khi ký.

Tự động hóa: Chuyển đổi file Word/Excel sang PDF tự động để lưu trữ lâu dài.

2. Công nghệ Sử dụng (Tech Stack)
Đây là bộ công nghệ chuẩn doanh nghiệp (Enterprise Standard), đảm bảo hiệu năng và tính bảo mật cao:

Backend (Lõi):

Java Spring Boot 3.x: Framework chính để xây dựng RESTful API.

Spring Security + JWT: Xử lý xác thực đăng nhập và phân quyền.

Spring Data JPA (Hibernate): Làm việc với Database.

Database (Lưu trữ):

PostgreSQL: Lưu thông tin người dùng, cấu trúc thư mục, log hoạt động (Metadata).

MinIO (hoặc Local File System): Lưu trữ file vật lý (PDF, Docx, Image). Lưu ý: Không lưu file vào Database, chỉ lưu đường dẫn.

Xử lý Tài liệu (Thư viện quan trọng):

Apache PDFBox / iText: Dùng để thao tác file PDF, chèn chữ ký, mã hóa chữ ký số.

JODConverter (hoặc LibreOffice Headless): Dùng để convert file Word/Excel sang PDF.

Frontend (Giao diện):

ReactJS / VueJS: Xây dựng giao diện người dùng tương tác cao.

PDF.js: Thư viện để hiển thị file PDF ngay trên trình duyệt.

3. Giải thích Yêu cầu Chức năng (15+ Chức năng & 1 Killer Feature)
Hệ thống được chia làm 5 Phân hệ (Module) tương ứng với 5 thành viên.

🌟 CHỨC NĂNG ĂN ĐIỂM (THE KILLER FEATURE)
Tên: Ký số & Kiểm tra Toàn vẹn Dữ liệu (Cryptographic Signing)

Yêu cầu:

Khi người dùng bấm "Ký", hệ thống không chỉ dán ảnh chữ ký lên PDF.

Hệ thống phải dùng Private Key (Khóa bí mật) của user để mã hóa chuỗi Hash của file.

Cơ chế kiểm tra: Nếu file sau khi ký bị ai đó sửa đổi nội dung (hack), hệ thống phải phát hiện ra ngay và hiện cảnh báo: "Chữ ký không hợp lệ / Tài liệu đã bị thay đổi".

👤 Phân hệ 1: Quản trị & Bảo mật (Admin & Security)
Yêu cầu đảm bảo tính chặt chẽ của hệ thống con người.
Tìm kiếm Nâng cao (Advanced Search): Tìm kiếm file theo nhiều tiêu chí kết hợp: Tên file + Người tạo + Khoảng thời gian upload.

Dashboard Thống kê: Biểu đồ hiển thị:

Tổng dung lượng đã dùng.

Số lượng văn bản chờ duyệt / đã duyệt.

Top nhân viên đóng góp tài liệu nhiều nhất.
Đăng nhập & Xác thực 2 lớp (Auth): Ngoài mật khẩu, gửi mã OTP về email nếu phát hiện đăng nhập lạ.

Quản lý Cơ cấu tổ chức: Tạo các Phòng ban (Phòng Nhân sự, Phòng Kế toán...). Gán nhân viên vào đúng phòng ban để sau này trình ký đúng quy trình.

Phân quyền vai trò (RBAC): Cấu hình chi tiết: Ai được xem? Ai được sửa? Ai có quyền Ký duyệt? (Ví dụ: Nhân viên chỉ được Upload, Sếp mới được Ký).

👤 Phân hệ 2: Quản lý Kho tài liệu (Document Storage)
Yêu cầu khả năng tổ chức dữ liệu khoa học.

Upload & Validate Đa định dạng: Cho phép upload Word, Excel, PDF. Hệ thống phải chặn file .exe (virus) và giới hạn dung lượng (ví dụ max 20MB).

Cây thư mục thông minh: Tạo thư mục cha/con không giới hạn cấp. Hỗ trợ kéo thả file để di chuyển giữa các thư mục.

Thùng rác & Khôi phục (Recycle Bin): File xóa sẽ vào thùng rác. Admin có thể khôi phục lại file đã xóa nhầm trong vòng 30 ngày.

👤 Phân hệ 3: Xử lý & Chuyển đổi (Converter & Viewer)
Yêu cầu khả năng xử lý file mạnh mẽ.

Convert tự động sang PDF: Khi User upload file .docx hoặc .xlsx, hệ thống phải tự động gọi service convert sang .pdf ở background để chuẩn bị cho việc ký (vì Word không ký số chuẩn được).

Trình xem tài liệu (Web Viewer): Tích hợp trình đọc PDF trên web. Hỗ trợ Zoom in/out, xoay trang, nhảy trang.

Chia sẻ an toàn (Secure Sharing): Tạo đường link chia sẻ file cho đối tác bên ngoài. Link này có thể cài đặt mật khẩu truy cập hoặc ngày hết hạn (ví dụ: link chỉ sống trong 24h).

👤 Phân hệ 4: Quy trình Trình ký (Workflow Engine)
Yêu cầu xử lý logic nghiệp vụ phức tạp.

Tạo luồng trình ký: User chọn file, chọn thứ tự người duyệt (Ví dụ: Nhân viên A -> Trưởng phòng B -> Giám đốc C).

Phê duyệt & Từ chối: Người duyệt nhận được thông báo.

Nếu Đồng ý: File chuyển sang người tiếp theo.

Nếu Từ chối: Phải nhập lý do, file quay ngược về người soạn thảo.

Nhật ký dấu vết (Audit Log): Ghi lại mọi hành động. Ví dụ: "Nguyễn Văn A đã xem tài liệu lúc 08:00", "Trần Văn B đã ký lúc 09:30". Dữ liệu này không được phép xóa.

👤 Phân hệ 5: Tiện ích & Quản lý phiên bản (Utilities)
Yêu cầu tăng trải nghiệm người dùng.

Quản lý Phiên bản (Versioning): Khi upload đè file mới lên file cũ, hệ thống tự động lưu thành v1.0, v1.1, v1.2. User có thể xem lại nội dung của phiên bản cũ bất cứ lúc nào.
1. Cơ chế hoạt động (Luồng xử lý)
Chúng ta sử dụng cơ chế Mã hóa bất đối xứng (Asymmetric Cryptography) với cặp khóa:

Private Key (Khóa bí mật): Dùng để KÝ. (Giữ bí mật tuyệt đối).

Public Key (Khóa công khai): Dùng để KIỂM TRA. (Công khai cho mọi người).

Quy trình Ký (Khi User bấm nút "Ký")
Bước 1: Hash file (Băm dữ liệu)

Hệ thống lấy toàn bộ nội dung file PDF, đưa qua thuật toán băm (SHA-256) để tạo ra một chuỗi ký tự duy nhất gọi là Message Digest (D).

Ví dụ: File A -> Hash: x8s9d... (Chỉ cần file A sửa 1 dấu chấm, Hash sẽ đổi thành k1l2m... ngay).

Bước 2: Mã hóa Hash (Tạo chữ ký)

Hệ thống dùng Private Key của User để mã hóa chuỗi Hash x8s9d... kia -> Kết quả là chuỗi Digital Signature (S).

Bước 3: Đóng gói PDF

Hệ thống dùng thư viện PDF để nhúng chuỗi (S) và Public Key (thường nằm trong Certificate X.509) vào metadata của file PDF.

Đồng thời, chèn thêm ảnh chữ ký .png vào trang PDF để hiển thị trực quan.

Quy trình Kiểm tra (Khi mở file hoặc bấm "Verify")
Hệ thống lấy Public Key trong file PDF để giải mã chuỗi Digital Signature (S) -> Ra được Hash gốc (D1).

Hệ thống tự tính toán lại Hash của nội dung file hiện tại -> Ra được Hash mới (D2).

So sánh:

Nếu D1 == D2: Chữ ký hợp lệ. File nguyên bản.

Nếu D1 != D2: File đã bị ai đó sửa đổi nội dung. Chữ ký bị vỡ (Invalid).
